# v0.1.4 UX Improvements — Design

**Date:** 2026-04-26
**Target release:** v0.1.3 + 0.1.4 (this spec covers what becomes v0.1.4)
**Status:** Approved (pending implementation)

## Background

After shipping v0.1.3 (per-save DB isolation), the user identified four UX feedback items from in-game testing:

1. The browser dashboard does not auto-refresh when a new in-game day ends.
2. A multi-save dropdown to view another save's data without changing the loaded save in Stardew.
3. By default, the dashboard does not auto-open in the browser when a save is loaded.
4. When a save is loaded mid-playthrough (not Spring Y1 D1), no warning is shown that the historical data will be partial.

**Item #2 was deferred** to a later release — the user wants to ship #1, #3, #4 first.
**Item #3 has already shipped** as a single-line config default flip in commit `089766b` on `main` (not yet released). It is included in v0.1.4 alongside #1 and #4.

This spec covers items #1 and #4 only, since #3 is already implemented.

## Goals

1. **Auto-refresh on day change** — The dashboard updates to reflect the latest day's data without the user manually reloading the page.
2. **Mid-history banner** — Users who install LaCompta partway through a playthrough see a one-time, dismissable banner explaining that historical data will start from the day of first tracking, not Spring Y1 D1.

## Non-goals

- Real-time push-based updates (SSE / WebSockets). Polling is sufficient given that data only changes once per in-game day.
- A multi-save selector dropdown (deferred per user, item #2).
- A configurable polling interval. YAGNI — fixed at 10 seconds.
- A reset/un-dismiss UI for the banner. Once dismissed, it stays dismissed for that save's DB.
- A history-reconstruction feature. We cannot infer past days from a save loaded mid-playthrough; the banner just informs the user.

## Item #1 — Auto-refresh on day change

### Mechanism: client-side polling

The dashboard JS polls `/api/farminfo` every **10 seconds**. The endpoint already returns `Year`, `Season`, `Day`, plus an `IsWorldReady` flag.

The client keeps a local tuple `lastSeen = (year, season, day)`. On each poll:

- If `IsWorldReady` is `false` (no save loaded), do nothing. Wait for the next poll.
- If `IsWorldReady` is `true` and the response tuple equals `lastSeen`, do nothing.
- If the tuple changed (or `lastSeen` was uninitialised), update `lastSeen` and re-run the dashboard's load functions to re-fetch and re-render data sections without a full page reload.

The change-detection naturally fires at `DayStarted` time (the new day after the player wakes up), which is the right UX moment — the previous day's totals have just been written by `TrackingService.OnDayEnding` and `SeasonSummaryService.OnDayEnding`, and the player is about to look at them.

A switch-of-save (return to title, load a different save) also changes the tuple → triggers a re-fetch automatically. Bonus side-effect of this design.

### Polling interval

10 seconds, hardcoded. On localhost the cost is negligible (one ~200-byte GET per 10s). Not promoted to `ModConfig` because nobody has asked and the trade-off space is narrow. If a complaint surfaces, it's a one-line change.

### Soft refresh, not full reload

The implementation re-runs the existing data-load functions in `app.js` (e.g., `loadOverviewPage()`, `loadCurrentPage()` or whatever the active-page entry-point is). It does **not** call `location.reload()`. This preserves:

- Active tab / page selection (Overview vs. Profitability vs. Sales Ledger vs. Legendary Fish vs. Season Compare)
- Scroll position
- Selected season filter, time-range buttons, etc.

### Pause when no save loaded

If `IsWorldReady` is `false`, we skip the data-fetch step entirely. The polling timer keeps running so we'll detect the next save load. No spam fetches against an empty DB.

|-------------------------------|----------------------------------------------------------------------------------------------|
| `LaCompta/Web/Assets/app.js`  | Add a polling loop that calls `/api/farminfo`, tracks the last-seen day tuple, calls into the existing load logic on change. |

No backend changes. `/api/farminfo` already returns the fields needed.

## Item #4 — Mid-history banner

### Trigger condition

At dashboard load, the client computes a `firstRecordDay` value (year, season, day) using this logic:

1. If `daily_records` table has at least one row: `firstRecordDay = MIN` of all rows by (year, season, day) — taking ordering into account so `(1, spring, 1) < (1, summer, 1)`. The backend exposes this via the new `/api/state` endpoint described below.
2. If `daily_records` is empty (save loaded but no day has ended yet): `firstRecordDay` falls back to the *current* in-game `(year, season, day)`.

The banner is shown if and only if **all** of:

- `firstRecordDay != (1, spring, 1)`, **and**
- The save's DB has no `mod_state` row with key `banner_mid_history_dismissed` set to `'1'`.

Saves that started fresh with LaCompta installed will have `firstRecordDay = (1, spring, 1)` once their first day is recorded, so the banner never fires for them. Saves loaded mid-playthrough will trigger the banner the first time the user opens the dashboard (and on subsequent opens until dismissed).

### Persistence: new `mod_state` table

A new tiny key/value table is added to the per-save DB schema:

```sql
CREATE TABLE IF NOT EXISTS mod_state (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
```

This table lives inside each save's `data/<SaveFolderName>.db` file. Two consequences:

- When a save is deleted in Stardew, the v0.1.3 reconciliation in `SaveCleanupService` removes the DB and the dismissed-flag with it. Re-creating a save with the same folder name would be a clean slate (highly unlikely scenario).
- Future banner / one-time-tip / per-save UX flags can reuse this table by adding new keys, with no schema migration.

The dismissed flag is stored as `INSERT OR REPLACE INTO mod_state (key, value) VALUES ('banner_mid_history_dismissed', '1')`.

### API

Two new endpoints on `ApiController`:

```
GET /api/state
→ {
    firstRecordYear: 1,
    firstRecordSeason: "spring",
    firstRecordDay: 5,
    midHistoryDismissed: false,
    isWorldReady: true
  }

POST /api/state/dismiss-mid-history
→ 200 OK (writes the row, no body)
```

`/api/state` is read at every dashboard load (and could later be polled if needed, but for now once-per-load is enough).

`POST /api/state/dismiss-mid-history` is one-shot; the JS calls it when the user clicks the banner's × button, and the JS hides the banner without waiting for a roundtrip (optimistic UI — the next page load will confirm via `/api/state`).

### Banner UI

- Renders at the top of `<div class="page-content">` or wherever the user-facing top-of-page content sits.
- Yellow-pale info background, single line of text, × button on the right.
- Copy: `"⚠️ LaCompta a démarré le suivi à partir du <Jour> <Saison> An <Année>. Les jours antérieurs ne sont pas dans les stats."` (with `<Jour>`, `<Saison>`, `<Année>` substituted at render time).
- Not modal, not blocking, doesn't interrupt the dashboard layout.

## Error handling

- `/api/farminfo` polling failure (network blip, server stopped between two polls): the JS swallows the error silently and tries again at the next interval. No user-visible spinner or error toast — failures are transient.
- `/api/state` failure on initial page load: the banner stays hidden by default. Better to skip the warning than block the dashboard. Log to browser console.
- `/api/state/dismiss-mid-history` failure: the banner has already been hidden optimistically by the JS. The flag isn't persisted, so it'll re-appear next page load. Acceptable — the user can dismiss again.
- Repository SQL failures: existing patterns. SQLite errors propagate as exceptions, the request returns 500.

## Testing

No automated test infrastructure (game-runtime required). Manual smoke test:

1. **Auto-refresh, fresh save:** start a new save (Spring Y1 D1). Open dashboard. Note the day shown. Sleep through the night in-game. Within ~10s of the new morning, the dashboard's day display should update without manual reload, and any newly-tracked income should appear in the totals.
2. **Auto-refresh, save switch:** with the dashboard open, return to title and load a different save. Within ~10s, the dashboard should switch to showing the new save's data automatically.
3. **Auto-refresh, idle:** with the dashboard open, return to title and stay there. The dashboard should stop fetching new data (verify in browser devtools that fetches still hit `/api/farminfo` but don't trigger data re-fetches).
4. **Banner, fresh save:** start a new save at Spring Y1 D1. Banner does NOT appear at any point.
5. **Banner, mid-play install:** load a save that's already at e.g. Summer Y2 D14. First time the dashboard opens, the banner appears with text mentioning "Summer An 2 Jour 14". Click ×. Banner disappears. Reload the page. Banner does NOT reappear.
6. **Banner persistence across sessions:** dismiss the banner. Quit Stardew. Restart. Load the same save. Open dashboard. Banner is still dismissed (no re-appearance).
7. **Banner per-save isolation:** dismiss banner on Save A. Load Save B (also mid-history). Banner appears for B. Dismiss it. Switch back to A — banner is still dismissed for A.

## Risks and mitigations

| Risk | Mitigation |
|------|------------|
| Polling thrashes the C# `HttpListener` event loop on slow machines. | 10s interval is well below any realistic load. The handler returns ~200 bytes of JSON. Worst case: skip a poll, next one catches up. |
| `IsWorldReady` false-positives between save loads (e.g., during the loading screen): briefly returns `IsWorldReady: false` then `true`. | Polling-based detection naturally tolerates this — we just skip a tick. |
| Banner copy/translation: the message is hardcoded French to match the mod's voice. | Out of scope for now; if i18n becomes a goal, swap to a string-table later. |
| `mod_state` table schema collides with a future feature wanting a different shape. | Key/value is the most generic shape; if a future feature needs structured data, store JSON as the value. |

## Out of scope (deferred)

- Item #2 — multi-save dropdown — explicitly deferred by the user.
- A "tracking start day" formal column in `season_summaries` (could replace the runtime MIN computation).
- An "I'm reading historical data, pause polling" toggle. YAGNI.
- Real-time DayEnding push (SSE) — only relevant if polling proves too slow, which it won't at the data-update cadence.
